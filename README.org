#+TITLE: aaronkuehler.com

Personal site and blog. Content is authored in Org mode, exported to
Markdown with [[https://ox-hugo.scripter.co/][ox-hugo]], built into a static site with [[https://gohugo.io/][Hugo]], and
published to GitHub Pages. Org is the source of truth — the generated
~content/~ and ~public/~ directories are git-ignored and never committed.

This supersedes the previous org-page toolchain; see
[[file:docs/adr/ADR-001-adopt-ox-hugo-and-hugo.md][ADR-001]] for the decision and the remaining migration work.

* Layout

| Path                    | Purpose                              |
|-------------------------+--------------------------------------|
| ~index.org~             | Home / about page                    |
| ~blog/*.org~            | Blog posts (one file per post)       |
| ~resume/~               | Résumé sources                       |
| ~org/ox-hugo-setup.org~ | Shared ox-hugo export settings       |
| ~build/export.el~       | Batch Org to Markdown export         |
| ~hugo.toml~, ~layouts/~ | Hugo configuration and templates     |
| ~static/~               | Files copied verbatim (for example, CNAME) |
| ~Dockerfile~            | Build + audit toolchain (Emacs, ox-hugo, TeX Live, Hugo, Chrome + pa11y) |

* Build and preview locally

The toolchain (Emacs + ox-hugo, TeX Live, Hugo, plus a headless Chrome
and pa11y for the accessibility audit) is packaged as a Docker image so
local builds match CI exactly. The image is amd64 -- Chrome has no Linux
arm64 build -- so on Apple Silicon it runs under emulation. With only
Docker installed:

#+begin_src sh
make image          # build the toolchain image once
make docker-build   # build the site into public/ inside the container
make shell          # or drop into a shell in the container
#+end_src

If you have Emacs, Hugo (extended), and LaTeX installed directly, the
host targets work too:

#+begin_src sh
make build          # build into public/, then audit it (WCAG AAA)
make a11y           # audit an already-built public/ on its own
make serve          # export + serve at http://localhost:1313
#+end_src

The first host export installs ox-hugo from MELPA into a project-local
~.elpa/~; the container has it pre-installed. ~make build~ ends with the
accessibility audit (pa11y at WCAG2AAA over the sitemap), which gates
~make deploy~; on the host that step also needs Node, pa11y-ci, and a
Chrome or Chromium, all of which the container already provides.

* Publishing

Pushing to ~master~ triggers the GitHub Actions workflow
(~.github/workflows/deploy.yml~). Its jobs run in order (~needs:~):
(re)build the toolchain image on GHCR when the ~Dockerfile~ or
~export.el~ changes, build the site inside that image, then deploy to
GitHub Pages. It uses only GitHub-owned actions plus the toolchain
container, so it avoids the account's block on third-party actions.

Pull requests additionally run ~.github/workflows/accessibility.yml~,
which builds the site and audits every page with pa11y at WCAG2AAA, so
accessibility regressions surface before merge.

In short: ~git push origin master~ publishes the site.

~make deploy~ (clean-build and force-push the output to ~gh-pages~) is a
break-glass fallback for publishing without Actions — using it requires
switching the Pages source to "Deploy from a branch" (~gh-pages~). ~master~
is the source of truth.

* Writing a post

Each post is a Hugo leaf bundle. Add ~blog/<name>.org~ with a header
modeled on an existing post:

#+begin_example
#+TITLE:       Post Title
#+DATE:        <2026-07-03 Fri>
#+DESCRIPTION: One-sentence summary.
#+SETUPFILE: ../org/ox-hugo-setup.org
#+EXPORT_FILE_NAME: index
#+HUGO_BUNDLE: <name>
#+HUGO_SLUG: post-slug
#+end_example

The published URL is ~/blog/YYYY/MM/DD/<slug>/~, taken from ~#+DATE~ and
~#+HUGO_SLUG~. If a post ever changes date or slug, add the old paths to
~#+HUGO_ALIASES~ (space-separated) so the previous URLs redirect instead
of breaking — this is how the historical URLs from the old site are kept
alive.

** Post images

Put a post's images in ~assets/blog/<name>/~ and reference them from the
Org file with a relative link:

: [[../assets/blog/<name>/diagram.png]]

At build time ox-hugo copies them into the post's bundle, so each image
is served under its own post's URL
(~/blog/YYYY/MM/DD/<slug>/diagram.png~) — namespaced per post, so two
posts can share an image filename without colliding.
